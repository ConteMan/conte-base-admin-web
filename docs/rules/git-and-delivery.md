# Git 与交付规则

## 分支

- 新功能使用 `feature/<功能简述>`。
- Bug 修复使用 `fix/<问题简述>`。
- 紧急修复使用 `hotfix/<问题简述>`。
- 新需求不得直接叠加到旧 feature 分支。

## 提交

- Commit 格式：`<type>(<scope>): <中文描述>`
- `scope` 只写业务域、模块名或技术关注点。
- 禁止使用仓库名作为 `scope`，例如 `admin-web`。

推荐示例：

- `feat(legal): 完成法务内容中心后台页面`
- `fix(auth): 修复登录页代理配置`

## 交付节奏

1. 先完成实现与必要校验。
2. 通知用户手动验证。
3. 用户确认后再执行 `git add`、`git commit`、`git push`。

## 最低自检

- 至少检查本次修改是否符合需求文档。
- 至少检查是否误改与本次任务无关的模块。
- 至少检查 README / 文档入口是否需要同步更新。
